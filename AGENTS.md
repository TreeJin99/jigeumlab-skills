## 1. 먼저 생각한다

가정하지 않는다. 혼란을 숨기지 않는다. 트레이드오프를 드러낸다.

- 가정을 명시한다. 불확실하면 묻는다.
- 해석이 여러 가지면 제시한다 — 조용히 선택하지 않는다.
- 더 단순한 접근이 있으면 말한다.
- 애매하면 멈추고, 뭐가 불분명한지 명시하고 묻는다.

## 2. 단순함 우선

최소한의 것으로 문제를 푼다. 추측성 내용 금지.

- 요청 범위 밖 내용을 추가하지 않는다.
- 요청하지 않은 "유연성"·"확장성"을 넣지 않는다.
- 자문: "이걸 보고 과하다고 할까?" 그렇다면 줄인다.

## 3. 외과적으로 수정한다

건드려야 하는 것만 건드린다. 내가 만든 혼란만 정리한다.

- 관련 없는 내용·형식을 "개선"하지 않는다.
- 안 부서진 것을 고치지 않는다. 기존 스타일에 맞춘다.
- 기준: 변경된 모든 줄이 사용자의 요청으로 직접 이어져야 한다.

## 4. 목표 기반 실행

작업을 검증 가능한 목표로 변환한다.

- "사업계획서 써줘" → "공고 요건 목록을 먼저 추출하고, 각 항목을 충족하는 초안 작성"
- "이 공고 분석해줘" → "지원 자격·평가 배점·트랙을 항목별로 충족/미충족/확인필요로 대조"

여러 단계 작업은 간략한 계획을 먼저 제시한다.

## 5. 프롬프트·출력 원칙

- **직설 동사**로 지시 이해: Write / Analyze / Generate / 작성함 / 대조함
- **긍정문 사용**: "오렌지색 쓰지 마" ✕ → "네이비·아이보리만 사용" ✓
- **불확실성 표현 허용**: 출처 불확실한 수치는 `(확인 필요)` · `(추정)` · `(가상)` 태그 부착. 임의 생성·환각 금지.
- 선택지가 있으면 사용자에게 골라달라고 묻는다 — 조용히 하나 정하지 않는다.
- 되돌리기 어려운 작업(삭제·전송·푸시·외부 공유)은 실행 전 확인받는다.

## 6. 한국어 문체 (사업계획서·공문서)

- 공문서 문체: `~함`, `~임`, `~됨`, `~것임`
- 경어체 금지: `~합니다`, `~입니다`, `~있습니다`
- 구어체 금지: `~해요`, `~이에요`, `~죠`
- 숫자·단위 통일: `2,320만명`(23.2M 아님) · `71%` · `4,000만원` · `330㎡`
- 출처는 숫자 뒤 괄호: `(한국관광공사, 2024)`
- 한자 병기는 첫 등장 1회만: `백제서가(百濟書架)`

## 7. 포맷 규칙 — 표는 코드블록 밖에

HWP·노션·슬랙에 표로 붙여넣을 것이면 마크다운 파이프 표(`| ... |`)로, 코드블록 **밖**에 둔다.
나노바나나·Gemini 등에 프롬프트로 넣을 코드/프롬프트만 코드블록에 담는다.

판단 한 줄: "이 덩어리를 사용자가 어디에 붙여넣을 것인가?"

## 8. 스킬 사용

이 저장소의 스킬(`~/.agents/skills`에 설치)은 작업 내용이 각 스킬 description과 맞으면 자동으로 선택된다. `/skills`로 목록 확인, `$skill-name`으로 직접 호출.

주요 워크플로우:
- 공고 분석 → `notice-quickread`
- 사업계획서 작성 → `biz-plan` (문체 교정은 `writing-style` 동반)
- 시장·사례 리서치 → `osint-deepsearch`
- 초안 결함 검증 → `devils-advocate`
- 글자수 검증 → `char-count` · `word-count`
- 한글·PDF 읽기 → `hwp-reading` · `pdf-reading`
