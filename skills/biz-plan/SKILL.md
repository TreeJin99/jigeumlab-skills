## 포맷 코덱 (프로그램별 역대 패턴)

> ⚠️ **포맷 파일은 과거 합격작 기반 패턴이다. 연도마다 섹션 구조·분량·항목이 바뀐다.**
> **항상 당해 공고문이 우선. 포맷 파일은 "이런 구조가 일반적이었다"는 참고용.**

사용법:
1. **공고문에서 올해 섹션/항목 직접 추출** (bizplan-researcher 역할)
2. 해당 프로그램 포맷 파일 로드 → 공고문과 **차이점 메모**
3. 차이가 있으면 공고문 기준으로 작성 (포맷 파일은 무시)
4. 차이가 없으면 포맷 파일 패턴 활용

## PSST 구조

| 섹션 | 핵심 질문 | 심사 관점 |
|---|---|---|
| **P — Problem** | 누가, 얼마나, 왜 겪는가? | 시장 규모·고통 구체성 |
| **S — Solution** | 어떻게 해결하나? 기존 대비 차별점은? | 기술성·실현가능성 |
| **S — Scale** | 어떻게 성장하나? 수익모델·KPI는? | 지속가능성·성장성 |
| **T — Team** | 왜 이 팀이 해낼 수 있나? | 전문성·실행력 증거 |

---

## 섹션별 작성 가이드

### P — 문제 정의
- 타깃 고객을 구체적으로 명명 (업종·규모·지역 포함).
- 고통의 빈도·비용·현재 해결책 한계를 수치로.
- 시장 규모: TAM → SAM → SOM → LAM 순으로. LAM(극초기 직접 전환 목표 시장)은 수치 근거(건수 × 전환율 × 단가) 반드시 명시.
- `[확인 필요: 시장 규모 출처]` 형식으로 미확인 수치 표기.

### S — 솔루션
- 핵심 기능 3개 이내로 압축. 나열 금지. (심사위원은 기능 나열을 기술 부재의 신호로 읽음)
- "기존 방식 vs 우리 방식" 비교 표 권장.
- 기술 스택·특허·독점 요소가 있으면 명시.

### S — 성장 (스케일)
- 수익모델: 과금 구조·단가·예상 고객 수.
- KPI 표 (아래 골격 사용).
- 채널: 획득 → 활성화 → 유지 경로 한 줄씩.

### T — 팀
- 핵심 인력별 한 줄 약력 (관련 경험만).
- 역할 분담표.
- 멘토·자문·파트너십 있으면 포함.

## 작업 절차

1. **공고문 분석** (bizplan-researcher)
   - 올해 섹션 목록, 항목명, 분량 제한 추출
   - 지원 자격·평가 항목·트랙 충족/미충족/확인필요 대조

2. **포맷 대조** (bizplan-writer 시작 전)
   - `references/format-{프로그램}.md` 로드
   - 공고문 섹션 vs 포맷 파일 패턴 비교
   - 차이점을 초안 상단에 `<!-- 올해 변경사항: ... -->` 로 메모

3. **초안 작성** (bizplan-writer)
   - **공고문 섹션 구조 우선**, 포맷 파일은 세부 표현/표 구조 참고
   - 미확인 수치는 `[확인 필요: {내용}]` 표기
   - 도식 작업 대상은 `[도식: {설명}]` 마커 삽입

4. 초안 완성 후 아래 체크리스트 점검
5. **클라이언트 확인** — notes.md 확인 필요 항목을 카카오톡으로 전달 → 답변 수신 → 사업계획서.md 반영 → `[확인 필요]` 마커 0건 달성 → 버전업
6. **PROGRESS.md에 "최종 QA 대기 — {파일명}" 표시**
7. 최종 QA 완료 확인 후 클라이언트 전달

### 작성 완료 체크리스트

- [ ] 모든 수치에 출처 또는 `[확인 필요]` 표기
- [ ] 개요(요약) 표 수치와 본문 각 섹션 수치 동기화 — OOO·[연도] 미완성 없음
- [ ] 자수·분량 제한 항목 확인 (초과 없음)
- [ ] "혁신적·게임체인저·패러다임" 등 빈 수식어 제거
- [ ] KPI·예산·추진일정 표 삽입
- [ ] 팀 역량 → 사업 실현가능성 연결 명시
- [ ] 공고 양식 항목과 섹션 1:1 대응 확인
- [ ] `[도식: {설명}]` 마커 삽입 완료 (나노바나나 작업 대상)

프로그램별 추가 체크리스트 → 해당 `references/format-{프로그램}.md` 고유 체크리스트 참조

## Constraints

- 공고문 섹션 구조 우선 — 포맷 파일과 충돌 시 공고문 기준으로 작성
- 미확인 수치는 `[확인 필요: {내용}]` 표기 필수 — LLM 자기평가로 수치 생성 금지
- 최종 제출 전 PROGRESS.md에 **"최종 QA 대기 — {파일명}"** 표시 필수
- 자동 제출 불가 — 최종 QA 완료 확인 후 클라이언트 전달

## Anti-Triggers

- 공고문 분석만 요청 ("이 공고 분석해줘") → bizplan-researcher 로 위임
- PSST 개념 설명 요청 ("PSST가 뭔지 설명해줘") → 개념 설명만, 작성 절차 실행 안 함
- 완성된 계획서 검토 ("이 계획서 검토해줘") → 작성 스킬 아님
- 도식 프롬프트 작성 ("나노바나나 프롬프트 써줘") → nanobanana 스킬로 위임
